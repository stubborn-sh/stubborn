# Feature 23: Command-Line Interface (CLI)

## What

A standalone command-line tool (`scc-broker`) that provides full access to the Spring Cloud
Contract Broker API. Users can register applications, publish contracts, record verifications,
manage deployments, check deployment safety, and administer webhooks вЂ” all from the terminal
without writing curl commands.

## Why

Without a CLI, users must:
- Construct raw curl/HTTP requests with correct headers, auth, and pagination
- Manually parse JSON responses to extract relevant information
- Write custom scripts for common workflows (publish в†’ verify в†’ deploy)

A CLI provides:
- Discoverable commands with `--help` for every operation
- Human-readable table/JSON output formatting
- Meaningful exit codes for CI/CD integration (e.g., `can-i-deploy` returns exit 2 if unsafe)
- Shell completion for command and option names
- Credential management via config file or environment variables

## How (High Level)

The CLI is a Spring Boot application using Picocli for command parsing. It wraps the
generated `broker-api-client` module вЂ” each CLI command maps to one API client call.
The CLI is packaged as a fat JAR (runs via `java -jar`) and optionally as a GraalVM
native image for instant startup.

```
User в†’ scc-broker app list в†’ Picocli parses в†’ ApplicationsApi.listApplications() в†’ Table output
User в†’ scc-broker can-i-deploy --app X --version 1.0.0 --env prod в†’ SafetyApi в†’ Exit code 0/2
```

## Commands

### Global Options

| Option | Env Var | Default | Description |
|--------|---------|---------|-------------|
| `--broker-url` | `SCC_BROKER_URL` | `http://localhost:8642` | Broker base URL |
| `--username` | `SCC_BROKER_USERNAME` | - | HTTP Basic username |
| `--password` | `SCC_BROKER_PASSWORD` | - | HTTP Basic password |
| `--output` | - | `table` | Output format: `table` or `json` |

## Business Rules

1. `--broker-url`, `--username`, `--password` can be provided via CLI options or env vars;
   CLI options take precedence
2. `can-i-deploy` exits with code 0 (safe) or 2 (unsafe); all other failures exit with code 1
3. `--output json` prints raw JSON from the API; `--output table` formats as a human-readable table
4. Paginated commands default to page=0, size=20
5. All write operations (register, publish, record, create, delete) print the created/modified
   resource to stdout on success
6. Errors print to stderr with the error code and message from the API

## Error Cases

| Scenario | Exit Code | Stderr |
|----------|-----------|--------|
| Broker unreachable | 1 | `Error: Cannot connect to broker at <url>` |
| Authentication failed | 1 | `Error: Authentication failed (401)` |
| Resource not found | 1 | `Error: <resource> not found (404)` |
| Validation error | 1 | `Error: <field>: <message> (400)` |
| Can-I-Deploy: unsafe | 2 | Prints result table, exits 2 |
| Unknown command | 1 | Picocli usage message |

## Acceptance Criteria

### Register and List Applications

**Given** the broker is running at `http://localhost:8642`
**When** I run `scc-broker app register order-service --owner team-commerce`
**Then** the exit code is 0
**And** stdout shows the registered application details

**When** I run `scc-broker app list`
**Then** the exit code is 0
**And** stdout shows a table with at least `order-service`

### Publish and List Contracts

**Given** application `order-service` exists
**When** I run `scc-broker contract publish order-service 1.0.0 --name get-order --content "request: ..." --content-type application/x-spring-cloud-contract+yaml`
**Then** the exit code is 0

**When** I run `scc-broker contract list order-service 1.0.0`
**Then** stdout shows a table containing `get-order`

### Record Verification

**Given** contracts exist for `order-service` v1.0.0
**When** I run `scc-broker verify record --provider order-service --provider-version 1.0.0 --consumer payment-service --consumer-version 2.0.0 --status SUCCESS`
**Then** the exit code is 0

### Can-I-Deploy Safety Check

**Given** `order-service` v1.0.0 has a successful verification and a staging deployment
**When** I run `scc-broker can-i-deploy --app order-service --version 1.0.0 --env staging`
**Then** the exit code is 0
**And** stdout shows "SAFE" or the safety result

**Given** no verifications exist for `order-service` v2.0.0
**When** I run `scc-broker can-i-deploy --app order-service --version 2.0.0 --env staging`
**Then** the exit code is 2

### Record Deployment

**Given** `order-service` v1.0.0 exists
**When** I run `scc-broker deploy record staging --app order-service --version 1.0.0`
**Then** the exit code is 0

### JSON Output

**When** I run `scc-broker app list --output json`
**Then** stdout is valid JSON matching the API response schema

### Environment Management

**When** I run `scc-broker env create staging --description "Pre-production" --display-order 2`
**Then** the exit code is 0

**When** I run `scc-broker env list`
**Then** stdout shows a table with `staging`

### Tag Management

**Given** `order-service` v1.0.0 exists
**When** I run `scc-broker tag add order-service 1.0.0 RELEASE`
**Then** the exit code is 0

**When** I run `scc-broker tag latest order-service RELEASE`
**Then** stdout shows `1.0.0`

### Webhook Management

**When** I run `scc-broker webhook create --url https://ci.example.com/hook --events CONTRACT_PUBLISHED,VERIFICATION_SUCCEEDED`
**Then** the exit code is 0

### Data Cleanup

**When** I run `scc-broker cleanup --keep-versions 5`
**Then** the exit code is 0
**And** stdout shows the cleanup result

### Version

**When** I run `scc-broker version`
**Then** stdout shows the CLI version string
